# ToxicJoin Threat Model

## Assets
- Sensitive warehouse data.
- DataHub metadata and governance classifications.
- SQL generated by AI agents.
- ToxicJoin policy configuration.
- Decision receipts and DataHub write-backs.

## Trust boundaries
1. AI agent to ToxicJoin API.
2. ToxicJoin to DataHub MCP/SDK.
3. ToxicJoin to DuckDB or warehouse.
4. Deterministic policy engine to optional LLM explanation layer.
5. Live mode to public replay mode.

## Adversaries and failure modes
- Agent produces an over-broad query unintentionally.
- Agent attempts to hide sensitive columns through aliases, CTEs, or nested queries.
- Metadata is missing, stale, or incorrect.
- LLM provides an unsafe explanation or recommendation.
- Query contains unsupported or mutating SQL.
- Replay is mistaken for a live system.
- Receipt or policy is modified after execution.

## Security invariants
- No query executes before a deterministic decision.
- BLOCK and uncertainty states never execute SQL.
- Optional LLM cannot override deterministic policy.
- Raw sensitive rows are not sent to an LLM.
- Unsupported SQL fails closed.
- Receipts contain hashes of relevant inputs and outputs.
- Replay is clearly labeled and cannot claim live mutations.
- Safe rewrites are verified after execution before release.

## Initial controls
- sqlglot AST parsing.
- Read-only query allowlist.
- Policy rules in versioned YAML.
- DataHub provenance attached to every finding.
- Minimum group-size verifier.
- Forbidden-column verifier.
- Input and output hashing.
- Secret scanning in CI.
- Synthetic demo data only.
