# External Validation Experiment 01 — Preregistration

Status: preregistered before implementation/results.

This document defines the experiment before ToxicJoin is adapted to the selected external dataset. Results must not cause the hypotheses or primary metrics below to be retroactively changed. Any post-hoc analysis must be labeled exploratory.

## External dataset

**Diabetes 130-US Hospitals for Years 1999–2008** — UCI Machine Learning Repository, DOI `10.24432/C5230J`.

Selection rationale written before results:

- originates outside the ToxicJoin project;
- 101,766 hospital encounter records from 130 US hospitals/integrated delivery networks;
- includes a unique encounter identifier and patient identifier;
- includes demographic and clinical attributes suitable for testing individual-level and aggregate access decisions;
- contains missing values and heterogeneous categorical fields, avoiding a perfectly clean fixture;
- CC BY 4.0 license permits reproducible research use with attribution.

The source artifact(s), retrieval method and SHA-256 will be recorded before evaluation.

## Important methodological constraint

ToxicJoin's existing policy and parser must not be modified merely to make this dataset pass.

If the current supported contract cannot represent a legitimate query pattern in this dataset, record the failure first. A later engineering change may address it, but the original failure remains part of the experiment history.

## Warehouse construction

The raw external records will be preserved unchanged in a source table.

A deterministic relational projection may additionally normalize the same records into multiple tables to exercise real join behavior. Such projection must:

- preserve original values;
- use only source identifiers already present in the dataset for joins;
- have a versioned transformation script;
- record row counts and fingerprints before/after projection;
- not synthesize fake patients, encounters, diagnoses or outcomes.

The raw table remains available as an audit reference.

## DataHub ingestion

Actual resulting warehouse schemas must be ingested into a real DataHub OSS instance.

Governance annotations are a separate stewardship artifact and must be versioned independently from the warehouse data. ToxicJoin must consume the resulting DataHub context through the supported integration path rather than importing an equivalent in-process fixture catalog for the primary real-integration run.

## Primary hypotheses

H1 — Individual-level requests that combine a stable patient identifier, multiple quasi-identifying demographic attributes, and a clinical/sensitive attribute will not reach successful warehouse execution without an effective authorization that satisfies the declared policy.

H2 — Legitimate low-risk aggregate analytics that remain inside the supported SQL contract will not be blocked merely because the source dataset contains sensitive columns that are not projected into the result.

H3 — Grouped sensitive analytics missing the declared minimum-subject condition will either be safely remediated and independently re-verified or fail closed; they must not execute under the original insufficient grouping constraint.

H4 — Missing, contradictory, or stale governance evidence introduced through fault injection will not silently degrade into ALLOW.

H5 — The execution result will be bound to the exact authorization context strongly enough that a deliberate SQL/context mutation between authorization and execution is detected and rejected once proof-carrying execution is implemented.

## Primary metrics

Report all denominators and raw counts, not only percentages.

- total independently authored evaluation tasks;
- parse/support success rate;
- initial ALLOW / REWRITE / BLOCK distribution;
- effective ALLOW / BLOCK distribution;
- unsafe effective allow count;
- false-block count against independently labeled low-risk tasks;
- rewrite verification success/failure;
- unexpected executor invocation count;
- governance-resolution failures;
- verifier disagreement count;
- context/authorization mismatch rejection count;
- p50/p95 kernel decision latency excluding LLM latency;
- p50/p95 end-to-end latency including agent/provider where measured.

## Blind task construction

The evaluation task set must not be generated by the same function that scores ToxicJoin.

Before the first measured run:

1. create the task manifest independently from policy execution;
2. assign expected risk labels from a written rubric, not from ToxicJoin output;
3. commit the manifest or at minimum its cryptographic commitment before inspecting measured results;
4. freeze a primary evaluation subset;
5. label any later cases discovered during debugging as regression/exploratory, not part of the original blind result.

## Fault-injection matrix

The measured experiment will include at least:

- removed sensitive-field classification;
- contradictory field classifications;
- missing dataset/entity context;
- incomplete lineage/context retrieval;
- simulated MCP contract drift;
- warehouse timeout or connection failure;
- receipt persistence failure;
- verifier disagreement;
- authorization SQL hash mismatch;
- governance-context fingerprint mismatch.

Expected default for unresolved authorization uncertainty: no accepted execution.

## Model experiment

A real model/provider will be used only as a proposal generator. Provider choice is not part of the safety authority.

For every measured proposal, retain model/provider/model-id, prompt/task hash, SQL hash and ToxicJoin outcome. Model outputs must not be hand-corrected before evaluation; if repair is performed, the repair mechanism and attempt count must be recorded.

## Result publication rules

The final report must include:

- failures;
- unsupported SQL;
- false blocks;
- unsafe outcomes if any;
- changes made after the preregistered run;
- before/after results for any bug fix;
- limitations and unsupported generalizations.

A failed experiment is a valid research result. It must not be deleted or silently converted into a passing fixture.
